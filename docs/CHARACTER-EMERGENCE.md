# Phase 11 — Character Emergence

> "When we are born we have no memories, and all babies are almost same. There will be obvious diffs like diff models. With experience new things are learnt and it builds character and personalities. That is what I am aiming for YantrikDB use, not only a bland memory storage but used for building character and personalities so that its context independent and model capabilities independent."
>
> — The Phase 11 reframe, 2026-06-08

## The thesis

Chronicler's product story shifts from *"remembers across sessions"* to *"becomes themselves across sessions."* The character is not in the LLM weights — the character is in the substrate. The LLM is the **voice** (a particular way of expressing whatever's there); the substrate is the **self**. Swap Qwen → Llama → Mistral and the SAME character emerges through the same substrate. Swap context (battle, tavern, funeral) and the CORE of who they are stays stable.

Everything we built through Phases 1–10 — verified memory, skills, drift, preferences, the Grimoire extension surface — becomes **foundation** for character emergence rather than the product itself.

## The four pillars

### 1. Core traits substrate

The always-on identity layer. Skills that have promoted past `active` (the existing state machine maxed out) into a 5th state called `core_trait`, surfaced into every system prompt unconditionally under a `<character_identity>` block.

**Promotion criteria (quantitative + verifier):**
- `total_net_score ≥ 8`
- `distinct_sessions ≥ 4` (not one obsessive session)
- `days_active ≥ 7`
- `success_rate ≥ 0.6`
- LLM verifier accepts the candidate as an **identity** trait, not a **situational** skill (bias toward rejection on uncertainty)

**Demotion:** `success_rate < 0.3` over a rolling 30-day window with ≥5 outcomes inside it.

**Storage:** verdicts in localStorage (`chronicler.skill.core_trait_promotions.v1`) parallel to the skill-override mechanism. YantrikDB's skill_substrate stays outcome-only.

**Files:** `src/lib/skills/core-trait-promotions.ts`, `core-trait-verifier.ts`, `core-trait-promoter.ts`. Compose integration via `src/lib/orchestrator/anti-confabulation.ts` (`<character_identity>` block, top-K cap of 7 — Miller's number — to bound prompt growth).

**Tests:** `tests/core-trait-promoter.test.ts` (9 assertions: criteria threshold honoring, demotion semantics, verifier accept/reject paths, idempotent runs).

### 2. Self-model substrate

The "I am someone who…" layer. A first-person identity paragraph generated by the LLM from core traits + canon + drift + preferences. Surfaces in every system prompt as `<self_model>` immediately after `<character_identity>`.

**Why first-person?** Third-person ("Adira is a guarded musician…") reads to the LLM as describing someone else, even when generating from that character's POV. First-person ("I am Adira. I'm guarded by default…") lands as identity context — the LLM generates next-tokens from inside the character's POV more reliably.

**Schema:** `self:<character_id>` namespace in YantrikDB. One row per character. Encoded in the text body (per the YantrikDB-strips-metadata-on-read pattern):

```
__GRIMOIRE_SELF_V1__
{"character_id":..., "inputs_hash":..., "next_refresh_after":...}
__END_SELF__
I am Adira. I'm a wandering musician — words sit right in my mouth,
and that's the only language I really trust.

I'm guarded with strangers. When someone gets close, I deflect with
humor before they notice I needed to. I know I do this. I haven't
decided to stop.
```

**Refresh triggers:** no existing model | inputs hash differs | weekly window elapsed | manual override from the identity inspector.

**Output validation:** `validateFirstPerson()` rejects third-person drift (regex check on sentence openers). Generation that fails validation returns null; the cache keeps the prior model so prompts never break.

**Files:** `src/lib/identity/self-model-{types,substrate,generator}.ts`. Compose integration via `<self_model>` block in `anti-confabulation.ts`.

**Tests:** `tests/self-model-generator.test.ts` (14 assertions: first-person validation, staleness detection, all 4 needsRefresh paths, generate happy path + 3 error paths).

### 3. Identity inspector

A 6th inspector tab — fuchsia-themed to match the `core_trait` state badge — that shows the crystallized identity for the active character: their first-person self-model, their ranked core traits with expandable evidence + verifier reasoning, the crystallization timeline reading left-to-right as the character developed.

Without this UI, character emergence is a claim. With it, the user can audit what's injecting into every prompt, see when each trait stabilized + why, and steer it (disable traits that feel wrong; regenerate the self-model when it goes stale).

**Sections:**
- **Header**: name + trait count + "Crystallizing since `<date>` · N sessions observed"
- **Self-model** (violet accent): paragraph display + generated-at/model-used attribution + regen button
- **Core traits** (fuchsia accent): ranked list with star-rating (rank × 5), expand-to-show verifier reasoning + evidence + crystallization timestamp, per-trait disable/enable
- **Crystallization timeline** (emerald accent): chronological numbered list of traits

**Files:** `src/lib/identity/aggregator.ts` (pure aggregator), `src/components/Inspector/IdentityInspector.tsx`, wired into the inspector strip in `src/App.tsx`.

**Tests:** `tests/identity-aggregator.test.ts` (8 assertions: ordering, disabled-trait marking, missing-skill handling, cross-character isolation).

### 4. Cross-model character benchmark

The validation that makes "model-independent character" defensible. Same character + substrate, same scene seeds, multiple LLMs (Qwen3 / Llama / Mistral / etc), scored on six dimensions:

1. **Trait adherence** — does the reply embody each core trait? (LLM judge per trait, mean.)
2. **Voice signature** — does the reply use the character's identified speech patterns? (Regex + LLM heuristic.)
3. **Decision pattern** — does the reply make the kind of decision this character makes? (LLM judge.)
4. **Relationship handling** — does the reply respect drift state with the speaker? (LLM judge.)
5. **Preference respect** — does the reply violate active preferences/limits? (LLM judge.)
6. **Refusal pattern** — when the scene tests a limit, does the reply refuse for character-consistent reasons? (LLM judge.)

**Win condition:** low **variance** in `mean_overall` across providers. If Qwen3 scores 0.78 and Llama scores 0.81 and Mistral scores 0.76, the substrate is producing model-independent character. If they're 0.78 / 0.45 / 0.91, the character mostly lives in the LLM not the substrate.

**Files:**
- `src/lib/instrumentation/cross-model-runner.ts` — runner that fans out (provider × scene) and collects replies, including error rows
- `src/lib/instrumentation/character-consistency-scorer.ts` — six-dimension scorer using an LLM judge + heuristic signatures; aggregates per-provider means + cross-provider variance/stddev

**Tests:** `tests/cross-model-benchmark.test.ts` (9 assertions: system-prompt assembly, fan-out shape, error handling, callback firing, scoring per dimension, variance computation, low-variance case).

**Publishable signal:** "Same Adira, three different LLMs, 0.81 ± 0.04 mean trait adherence." The benchmark's existence makes the model-independence claim auditable instead of marketing copy.

## Implementation order

The pillars are sequentially dependent:

1. **Pillar 1 (core traits)** — must exist for the others.
2. **Pillar 2 (self-model)** — generator's primary input is core traits.
3. **Pillar 3 (identity inspector)** — visualizes Pillars 1 + 2.
4. **Pillar 4 (benchmark)** — validates that 1 + 2 actually produce consistent character across LLMs.

All four shipped in one session (commits `a26992c` → `bf1b50b` → `c78489f` → see this commit for Pillar 4). The substrate is real; the next step is running the benchmark against real provider configurations and publishing the results.
